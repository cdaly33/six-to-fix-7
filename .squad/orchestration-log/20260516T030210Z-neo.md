## Current State Assessment

### SignalR `/hubs/audit-run` — Fully Implemented

Both `AuditOrchestrator` and `SkillRunner` actively publish events. `AuditDetail.razor` consumes via `HubConnection`.

**Two open defects regardless of architecture choice:**
1. `JoinAuditRun` does NOT validate tenant ownership — cross-tenant exposure risk
2. Blazor client passes no JWT token via query string (ADR-004 specifies required)

### Polling Alternative — Backend Shape

Minimal polling API already available:
```
GET /api/audit-runs/{id}   (returns status, timestamps)
```

For richer progress (recommended if removing hub):
```
GET /api/audit-runs/{id}/status   (returns completedSkillCount, currentSkillName, etc.)
```

Both read from existing `AuditRuns` + `SkillRuns` tables — no schema change. At 5-second interval with 2–5 users: ~10 queries/minute, negligible cost.

**Cost to remove SignalR:** ~1 day backend + frontend work (delete hub, notifier, modify orchestrator, add polling endpoint).

---

## Infrastructure Assessment

### pgBouncer

Not justified at 2–5 users (2–5 circuits max vs. 100+ connection pools needed). However, already designed around correctly (advisory locks, port split). **Recommendation: Leave in place** — low-value churn to remove.

### Background Channel\<HubSpotEvent\>

Justified. Decouples HubSpot latency from publish response. **Recommendation: Keep as-is.** Existing `hubspot_sync_queue` DB table available for future upgrade to outbox pattern if reliability becomes critical.

### Redis

Not present in stack. Not needed at this scale (no distributed cache, no scale-out SignalR, PostgreSQL advisory locks cover reviewer lockout). **Recommendation: Do not add.**
