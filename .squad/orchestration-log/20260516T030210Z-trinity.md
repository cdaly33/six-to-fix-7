## Payload

Task: Assess SignalR Hub implementation in `AuditDetail.razor` and recommend between push vs. polling architecture for 2–5 concurrent users.

---

## Findings

### Blocking Issues (SignalR Path)

1. **Broken Auth:** `AuditRunHubClientFactory.Create()` builds `HubConnection` with no JWT token. ADR-004 requires `access_token` query string. `[Authorize]` will reject all connections in production.

2. **Missing Tenant Check:** `JoinAuditRun(auditRunId)` has no tenant ownership validation. Any authenticated user can join any group — cross-tenant exposure risk.

### Architecture Cost (SignalR Path)

- Creates **second WebSocket connection** per user tab, on top of Blazor Server circuit
- 2 persistent connections per user = operational complexity (sticky sessions, scaling)
- Zero UX benefit at 2–5 users

### Polling Alternative

Because Blazor Server polls on the server, no extra browser HTTP or WebSocket needed:
- `PeriodicTimer` in component calls `IAuditOrchestrator.GetAuditRunAsync()` every 3 seconds
- ~1.7 DB queries/second during active runs — negligible against PostgreSQL Flexible Server
- Zero auth gap, zero tenant check gap

---

## Recommendation

**Remove dedicated hub for now. Use `PeriodicTimer` polling in `AuditDetail.razor`.**

Keep `IRealtimeNotifier` / `AuditRunHubNotifier` abstraction in Infrastructure — clean architecture, costs nothing dormant. Revisit if load grows or sub-second latency becomes a real requirement.

If team wants to keep SignalR: fix two defects above before merge.
