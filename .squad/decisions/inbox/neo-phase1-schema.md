## Decision: Global Query Filter Pattern (IsResolved Guard)

EF Core's `HasQueryFilter` is evaluated at model build time (once per DbContextOptions), not per request. The filter lambda captures `_tenantContext` (an interface reference) and EF Core evaluates the property at query time.

**Pattern used:**
```csharp
builder.Entity<Client>().HasQueryFilter(e => !_tenantContext.IsResolved || e.TenantId == _tenantContext.TenantId);
```

The `!_tenantContext.IsResolved` guard ensures:
- When `IsResolved = false` (background jobs, migrations, SuperAdmin), the filter passes all rows through
- When `IsResolved = true` (normal HTTP requests), the filter enforces TenantId equality
- Tenant table is NOT filtered (SuperAdmin must enumerate all tenants)

This implements ADR-002 and ADR-007.

## Decision: Migrations Deferred

EF Core migrations are NOT run in Phase 1. The design-time factory (`SixToFixDbContextFactory`) uses a `DesignTimeTenantContext` with `IsResolved = false` so no query filters are applied during DDL generation.

Run migrations after all Phase 1 files are committed and the solution builds cleanly:
```
dotnet ef migrations add InitialCreate -p src/SixToFix.Infrastructure -s src/SixToFix.Web
```

Connection string must use port 5432 (direct PostgreSQL, sf_admin role) for migrations — not pgBouncer port 6432.

## Decision: EFCore.NamingConventions Package Required

`builder.UseSnakeCaseNamingConvention()` requires the `EFCore.NamingConventions` NuGet package. **Morpheus must add this to `src/SixToFix.Infrastructure/SixToFix.Infrastructure.csproj`:**

```xml
<PackageReference Include="EFCore.NamingConventions" Version="9.*" />
```

Without this package the project will not compile.

## Decision: Entity Model vs. ER Diagram Reconciliation

The entity model implemented in Phase 1 uses the entity structure specified in the task prompt (15 entities: Tenant, User, Client, Audit, AuditRun, CategoryConfig, SkillRun, CategoryResult, CategoryResultVersion, Policy, PolicyFlag, CouncilSession, HubSpotSyncQueue, BlobReference, ReviewerLockout). This differs from the ER diagram planning artifact which has a different table grouping. The task prompt entities represent the evolved, authoritative implementation model.

---

## Decision: User Domain Entity vs. ApplicationUser

- `User` (Domain entity) — represents the domain concept, used for service layer logic
- `ApplicationUser` (Infrastructure entity, extends `IdentityUser<Guid>`) — the EF Core / ASP.NET Identity entity stored in the database
- `DbSet<User>` is NOT added to the DbContext; the Identity tables manage user persistence via `ApplicationUser`
- Services that need user data should use `ApplicationUser` via `UserManager<ApplicationUser>` or query Identity tables
